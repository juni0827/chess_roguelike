# DEV-5 · 체스 로그라이크 핵심 화면 IA와 와이어프레임 정리

## 1) Goal

- DEV 팀이 Android 클라이언트 구현 시 **화면 구조(IA), 정보 우선순위, 상태 처리 규칙**을 동일하게 이해하도록 기준 문서를 고정한다.
- CEO-13 핸드오프 원칙에 맞춰 다음 담당자가 바로 이어받을 수 있도록, 화면별로 **결정사항 / 남은 결정사항 / 다음 액션**을 함께 남긴다.

---

## 2) Inputs

### 2-1. 상위/연계 이슈 입력

- 상위 조율 이슈: `CEO-13` (CEO↔DEV 핸드오프 기준)
- 연계 이슈:
  - `DEV-4` 핵심 UX 플로우/화면 구조 초안
  - `DEV-6` 전투 HUD 정보 우선순위
  - `DEV-7` 메타 진행/보상 선택 UX
  - `DEV-3` 비주얼 레퍼런스/스타일 방향

### 2-2. 제품 전제

- 플레이 루프: `Main → Game(라운드 전투) → Upgrade(보상 선택) → Game`
- 런 종료 조건:
  - 패배: 플레이어 킹 사망
  - 승리: 최대 라운드(현재 5) 완료
- 플랫폼 우선순위: Android 1차, Desktop은 동일 IA 참조

---

## 3) Core IA

## 3-1. 전역 화면 지도

```text
[Main]
  ├─ 새 게임 시작 → [Game]
  ├─ 이어하기(세이브 존재 시) → [Game]
  ├─ 모드 가져오기 (in-place)
  └─ 언어 변경 (in-place)

[Game]
  ├─ 라운드 진행/전투
  ├─ 라운드 승리 이벤트 → [Upgrade]
  ├─ 런 패배/승리 다이얼로그 → Main 복귀 or 재시작
  └─ 런 중단(종료 확인) → Main

[Upgrade]
  ├─ 업그레이드 선택 확정 → [Game] 복귀
  └─ 취소/이탈 시 → 선택 재요청(복귀 시점에서 다시 진입)
```

## 3-2. 화면별 정보 우선순위

- **Main**: 진입 액션(시작/이어하기) > 런 상태 요약 > 시스템 액션(모드/언어) > 규칙 안내
- **Game**: 전투 보드 > 턴/목표 HUD > 진행 메타(라운드/점수/위협도) > 보조 액션
- **Upgrade**: 선택 카드 목록 > 선택 확정 > (복귀 전) 선택 검증

## 4) Screens

## 4-1. Main 화면 (허브)

### IA 블록

1. 헤더: 게임 타이틀/서브타이틀
2. 핵심 액션: `새 게임 시작`, `이어하기`
3. 시스템 액션: `모드 가져오기`, `언어`
4. 런/프로필 요약: 재화, 최고점수, 모드 수, 현재 언어, 이어하기 가능 여부
5. 규칙/가이드 패널

### 저충실도 와이어프레임

```text
┌────────────────────────────────────┐
│ [Title] Chess Roguelike            │
│ [Subtitle] 체스 기반 로그라이크     │
├────────────────────────────────────┤
│ [Primary CTA] 새 게임 시작          │
│ [Secondary CTA] 이어하기            │
│ [System] 모드 가져오기 / 언어       │
├────────────────────────────────────┤
│ 재화/최고점수/모드/언어/런 상태      │
├────────────────────────────────────┤
│ 규칙 요약/플레이 안내                │
└────────────────────────────────────┘
```

### 개발 전달 포인트

- `이어하기`는 저장 데이터/콘텐츠 호환성 확인 후 활성화.
- 모드/언어 변경은 Main in-place로 즉시 반영.
- 런 상태 문구는 단순 유무가 아니라 차단 사유(호환 불가 포함)를 노출.

## 4-2. Game 화면 (전투)

### IA 블록

1. 상단 HUD
   - 라운드 진행(`n/max`), 점수, 턴 상태, 위협도, 현재 목표
2. 전투 보드
   - 8x8 보드, 선택 기물, 유효 이동 강조
3. 하단 정보 패널
   - 아군 기물 수, 적 기물 수, 포획 수
4. 액션 영역
   - 조건부 버튼: `이중 이동 건너뛰기`
   - 상시 버튼: `게임 종료`

### 저충실도 와이어프레임

```text
┌────────────────────────────────────┐
│ Round n/max | Score                │
│ [Progress]                         │
│ Turn Status | Threat               │
│ Objective                          │
├────────────────────────────────────┤
│                                    │
│            [Board 8x8]             │
│                                    │
├────────────────────────────────────┤
│ 내 기물  포획 수  적 기물           │
│ [조건부] 이중 이동 건너뛰기          │
│ [상시] 게임 종료                    │
└────────────────────────────────────┘
```

### 개발 전달 포인트

- 승리/패배 다이얼로그에서 `재시작`과 `메인 메뉴`를 모두 제공.
- 업그레이드 미선택 상태가 남으면 `Upgrade`를 다시 띄워 루프 일관성 유지.
- 턴 상태와 목표 문구는 게임 상태머신과 1:1 매핑(임의 텍스트 금지).

## 4-3. Upgrade 화면 (보상 선택)

### IA 블록

1. 타이틀/설명
2. 업그레이드 카드 리스트(3개 기본)
3. 카드 선택 후 확인 다이얼로그
4. 확정 시 `Game` 복귀

### 저충실도 와이어프레임

```text
┌────────────────────────────────────┐
│ [Title] 업그레이드를 선택하세요      │
│ [Subtitle] 다음 라운드를 준비합니다  │
├────────────────────────────────────┤
│ [Card A] 이름 + 설명                │
│ [Card B] 이름 + 설명                │
│ [Card C] 이름 + 설명                │
├────────────────────────────────────┤
│ 탭 → 확인 다이얼로그 → 선택 확정     │
└────────────────────────────────────┘
```

### 개발 전달 포인트

- 전달받은 업그레이드 목록이 비면 즉시 취소 결과로 종료.
- 선택 취소 시 화면 유지, 확정 시에만 결과 반환.
- 능력 부여형 업그레이드는 Game에서 대상 기물 지정 단계가 추가됨.

---

## 5) States (Empty / Loading / Error / Success)

## 5-1. Main

- **Empty**: 저장된 런 없음 → `이어하기` 비활성 + 안내 문구
- **Loading**: 앱 초기 구동/콘텐츠 읽기 중 (스플래시 또는 비동기 준비 상태)
- **Error**: 저장/콘텐츠 호환 실패, 모드 import 실패 토스트 노출
- **Success**: 런/프로필 요약 정상 렌더

## 5-2. Game

- **Empty**: 유효 런 상태 없음(`state == null`) → 화면 종료 후 Main 복귀
- **Loading**: 라운드 전환/보드 렌더 준비(현재는 동기 처리)
- **Error**: 잘못된 선택(아군 기물 아님 등) 토스트 + 입력 유지
- **Success**: 턴 진행, 승패 판정, 업그레이드 오퍼 처리

## 5-3. Upgrade

- **Empty**: 업그레이드 목록 없음 → `RESULT_CANCELED` 반환
- **Loading**: 업그레이드 후보 계산/전달 대기(현재는 진입 전 완료 가정)
- **Error**: 확인 실패/잘못된 ID(방어적으로 취소 처리)
- **Success**: 업그레이드 1개 선택 확정 후 Game으로 복귀

---

## 6) Handoff Criteria

아래 항목이 충족되면 DEV-5 핸드오프 완료로 본다.

1. Main/Game/Upgrade 3개 화면에 대해 IA 블록과 와이어프레임이 문서화되어 있다.
2. 각 화면의 Empty/Loading/Error/Success 상태와 복구 동선이 정의되어 있다.
3. Game ↔ Upgrade 전이 조건(라운드 승리, 미선택 복귀, 능력 부여형 후속 선택)이 명시되어 있다.
4. 연계 이슈(DEV-4/6/7/3)와의 의존 지점이 문서에 표시되어 있다.
5. 다음 구현 담당자가 즉시 착수 가능한 수준으로 결정/미결정/다음 액션이 정리되어 있다.

## 7) Handoff Snapshot (CEO-13 규칙 반영)

### 현재 결정 사항

- 코어 루프는 `Main → Game → Upgrade`로 고정.
- 핵심 화면 범위는 Main/Game/Upgrade 3개로 제한.
- 상태 분류는 화면별 Empty/Loading/Error/Success 4분류를 공통 적용.

### 남은 결정 사항

- 로딩 상태를 사용자에게 어느 수준으로 노출할지(UI 스켈레톤 vs 인디케이터) 최종 확정 필요.
- Upgrade 카드 시각 우선순위(희귀도, 태그, 프리뷰 아이콘) 디자인 가이드 확정 필요.
- Game HUD의 위협도 계산 근거를 규칙화할지(현재 라운드 기반 휴리스틱) 결정 필요.

### 막힌 점 / 필요한 입력

- 필요 입력: DEV-3의 비주얼 스타일 토큰(색/타이포/카드 밀도) 확정본.
- 의사결정 주체: CEO(브랜드/방향), DEV(기술 타협안), 공동 리뷰.

### 다음 액션

1. DEV-6과 맞물려 Game HUD 문구/밀도 최종 축소안 확정.
2. DEV-7과 맞물려 Upgrade 카드 메타 정보(희귀도/시너지 태그) 노출 레벨 결정.
3. 구현 착수 전, 본 문서를 기준으로 안드로이드 레이아웃 점검 체크리스트 생성.
