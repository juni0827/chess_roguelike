# CEO-11: 체스 로그라이크 게임 콘셉트 후보 비교와 방향 제안

## 1) Problem
체스 룰의 즉시 이해 가능성은 살리면서도, 로그라이크 장르의 반복 플레이 동기(매판 다른 빌드/선택/리스크)를 함께 제공할 수 있는 제품 방향을 빠르게 확정해야 한다.  
현재 DEV 하위 이슈(DEV-3~10)가 병렬 진행 중이므로, CEO 단계에서 **MVP 핵심 판타지·포함/제외 범위·우선순위**를 고정하지 않으면 구현 비용이 급격히 증가한다.

## 2) User Story
- **전략 지향 플레이어**로서, 익숙한 체스 문법 위에서 매 턴 의미 있는 선택을 하고 싶다.
- **짧은 세션 플레이어(모바일 중심)**로서, 10~20분 단위 런에서도 성장감과 다음 판 동기를 얻고 싶다.
- **콘텐츠 확장 운영 관점(팀/프로젝트)**에서, 신규 기물/유물/이벤트를 데이터 중심으로 추가 가능한 구조를 원한다.

## 3) 콘셉트 후보

### A. 정통 체스 전술 강화형
- 설명: 체스 보드 전술/퍼즐성을 최우선. 로그라이크 요소는 최소화(맵 분기·보상 선택 정도).
- 강점: 학습 곡선 명확, 구현 단순, 밸런싱 리스크 낮음.
- 약점: 반복 플레이 동기(빌드 다양성) 약함.

### B. 카드/스킬 덱빌딩 결합형
- 설명: 기물 이동 외에 카드 사용으로 액션을 확장(즉시 이동/버프/턴 조작 등).
- 강점: 폭발적인 시너지/리플레이성.
- 약점: UX 복잡도·밸런싱·콘텐츠 제작비 모두 높음.

### C. 메타 진행/성장 중심형
- 설명: 런 내 다양성보다 런 외 성장(해금, 영구 강화, 클래스별 트리) 비중이 큼.
- 강점: 장기 잔존율 설계에 유리.
- 약점: 초반 핵심 재미가 약하면 이탈률 상승.

### D. 체스 코어 + 유물 시너지 하이브리드형 (추천)
- 설명: 체스 이동/포지셔닝은 단순·명료하게 유지하고, 런 변주는 유물/축복 중심으로 설계.
- 강점: 구현 난이도 대비 차별성·리플레이성 균형.
- 약점: 유물 풀 설계 품질이 곧 게임 품질이 되는 구조.

## 4) 비교 매트릭스 (5점 만점)

| 평가 기준 | A 전술 강화형 | B 덱빌딩형 | C 메타 중심형 | D 하이브리드형 |
|---|---:|---:|---:|---:|
| 구현 비용(낮을수록 고득점) | 5 | 2 | 3 | 4 |
| 구현 난이도(낮을수록 고득점) | 5 | 2 | 3 | 4 |
| 차별성/화제성 | 3 | 5 | 3 | 4 |
| 모바일 세션 적합성 | 4 | 3 | 4 | 5 |
| 라이브 확장성(콘텐츠 추가 용이성) | 3 | 4 | 4 | 5 |
| **총점** | **20** | **16** | **17** | **22** |

## 5) 외부 레퍼런스 기반 시사점
- 체스 변형 로그라이크의 흥행/인지 사례(예: *Shotgun King*, *Pawnbarian*)는 검증됐지만, 공통적으로 "규칙은 단순, 변주는 명확"한 구조가 반응이 좋다.
- 최근 시장은 로그라이크/덱빌딩 과밀 경쟁이 심화되어, 초반 MVP에서 시스템 중첩(체스+카드+복수 자원)을 과하게 늘릴수록 onboarding 실패 확률이 커진다.
- 따라서 초기 제품은 "체스 코어 문법 + 가벼운 시너지 레이어"로 시작하고, 반응 지표 기반으로 카드 시스템을 확장하는 단계 전략이 유리하다.

## 6) 최종 제안 방향

### 최종안: **D. 체스 코어 우선 하이브리드**

#### 방향 원칙
1. 플레이어의 첫 3분 내 이해 가능해야 한다.
2. 한 런의 주요 의사결정은 "이동/교환"과 "유물 선택" 두 축으로 제한한다.
3. 카드형 능동 액션은 MVP에서 제외하고, 유물의 수동/조건부 트리거로 대체한다.

#### MVP 포함 범위 (In)
- 6x6 또는 8x8 단순 보드 전투 룸
- 기본 기물 세트 + 적 유닛 변형 소수
- 유물 30~40종 (공격/방어/경제/리스크)
- 맵 분기 2~3갈래
- 전투 보상: 유물 선택 / 회복 / 통화
- 런 외 메타: 최소 해금(초기 기물/유물 풀 확장)

#### MVP 제외 범위 (Out)
- 플레이어 덱 편집형 카드 시스템
- 실시간 전투 모드
- 복잡한 클래스별 스킬 트리
- 과도한 룸 타입(미니게임/장기 이벤트 체인)

## 7) 단계별 실행 로드맵

### Phase 1 (지금~MVP 데모)
- D안 기준 핵심 루프 고정: 전투 → 보상 선택 → 분기 이동 → 보스.
- 유물 데이터 스키마와 밸런싱 가드레일 수립.
- DEV-8/9/10에서 엔진·상태·데이터 구조를 D안 제약에 맞춰 잠금.

### Phase 2 (MVP 검증 이후)
- 텔레메트리 기반으로 저성과 유물군 교체.
- 소량의 능동형 액션(의사 카드) A/B 테스트.

### Phase 3 (라이브 확장)
- 카드형 시스템 또는 클래스 확장을 실험적 모드로 투입.
- 시즌형 콘텐츠(새 유물군/보스 규칙) 운영.

## 8) Acceptance Criteria
- [ ] CEO가 단일 추천안(D)을 승인하고, 차선안/제외안을 문서로 고정한다.
- [ ] DEV 팀이 바로 구현 가능한 입력값(루프, 스키마, HUD 우선순위, 상태 전이)을 확보한다.
- [ ] MVP In/Out이 명시되어 범위 확장 요청 시 기준 문서로 사용 가능하다.
- [ ] CEO-13 핸드오프 규칙에 맞춰 남은 결정/막힌 점/다음 액션이 기록된다.

- 아트/디자인 레퍼런스(DEV-3, DEV-5)에서 정보 밀도/가독성 제약 반영

## 10) Handoff Snapshot (CEO → DEV)

### 현재 확정 사항
- 추천안은 D(체스 코어 + 유물 시너지 하이브리드)로 고정.
- MVP는 카드형 능동 액션 없이도 재미 검증 가능한 범위로 제한.
- 우선 지표는 "초반 이해도", "런 완주율", "유물 선택 다양성".

### 남은 결정 사항
1. 보드 크기 최종 확정(6x6 vs 8x8)
2. 유물 초기 풀 정확한 수량(30/36/40)
3. 런 시간 목표(평균 12분 vs 18분)

### 막힌 점 / 필요한 입력값
- DEV-6/7에서 HUD 정보 우선순위 시안 필요(턴 정보, 유물 트리거, 위험 예고).
- DEV-10에서 유물 트리거 DSL(조건/효과 표현 범위) 초안 필요.
